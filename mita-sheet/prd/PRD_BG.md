## РАЗДЕЛ 1 — ВИЗИЯ, ЦЕЛИ, НЕ-ЦЕЛИ

### 1.1 Визия

**MITA Дневен Бюджетен Лист** е Google Sheets шаблон, който показва на
всеки българин едно единствено число всеки ден:

> **"Днес можеш да похарчиш €X."**

За разлика от всички останали бюджетни таблици, MITA не просто показва
червена клетка когато си похарчил повече. Тя автоматично преразпределя
оставащите дни на месеца — взима малко от бъдещите дискреционни разходи
и балансира бюджета, без потребителят да прави нищо ръчно.

Продуктът е **MVP валидация** на концепцията преди разработката на
мобилно приложение. Целта е да провери дали хората в България ще платят
за ежедневен автоматичен бюджет и да събере начален капитал.

---

### 1.2 Проблем

Българинът с нетна заплата от €800–1,200 знае колко получава на месец.
Но не знае колко може да похарчи **днес** — след наем, комунални,
застраховки, хранителни стоки и вноски.

Съществуващите решения имат три проблема:

| Проблем | Последствие |
|---|---|
| Само месечно планиране | Потребителят губи контрол след първата неочаквана покупка |
| Ръчно коригиране при превишаване | Сложно, досадно, повечето се отказват |
| Не отчитат реалната цена на живота в БГ | Шаблони, правени за западна Европа или САЩ |

---

### 1.3 Решение (накратко)

Таблица, която:

1. **Онбординг за 3 минути** — потребителят въвежда доход, фиксирани
   разходи и спестовна цел
2. **Автоматично определя клас на дохода** — 5 нива специфично за
   българската икономика в EUR
3. **Генерира категории с реалистични суми** — базирани на реални цени
   от Kaufland и Lidl България
4. **Разпределя бюджета по 30/31 дни** — с логика за типа разход
   (фиксиран / разпределен / групиран)
5. **Автоматично преразпределя при превишаване** — без намеса на
   потребителя, с пълен лог на промените
6. **Показва напредъка към цел** — спестявания, авариен фонд

---

### 1.4 Целева аудитория

**Основен сегмент (v1):**

| Характеристика | Описание |
|---|---|
| Възраст | 22–45 години |
| Местоположение | България (фокус: София, Пловдив, Варна, Бургас) |
| Доход | €600–2,500 нето/месец |
| Технологична грамотност | Ползват Google Sheets или Excel поне веднъж |
| Болка | Знаят колко получават, но не знаят колко могат да харчат ден за ден |
| Мотивация | Искат да спестят, но досегашните таблици са твърде сложни |

**Вторичен сегмент:**

- Семейства с 2 дохода (нужда от обща бюджетна картина)
- Фрийлансъри с нередовен доход (нужда от гъвкав месечен план)

### 1.5 Ценностно предложение

**За купувача на €12–15:**

> *"Не е приложение, не е абонамент. Веднъж купуваш, веднъж копираш в
> Google Drive, и таблицата работи за теб целия месец. Когато превишиш
> бюджета за ден, тя сама разпределя оставащите дни — ти не правиш
> нищо."*

**Разлика от безплатните алтернативи:**

| Безплатни шаблони | MITA |

| Английски интерфейс | Изцяло на български |
| Общи проценти | Настроени за доходите в България |

### 1.8 Ограничения и допускания

- **Потребителят има Google акаунт** — без него Google Sheets не работи
- **Разходите се въвеждат ръчно** — 30 секунди на транзакция;
  това се приема като feature (осъзнато харчене), не bug
- **Таблицата е за един месец** — при нов месец потребителят прави
  нова копия или натиска "Нов месец" (описано в Раздел 11)
- **Цените от Kaufland/Lidl се обновяват периодично** — не в реално
  време, а при всяка нова версия на шаблона или при ръчно обновление
- **Таблицата не замества финансов съветник** — само помага за
  ежедневен контрол

## РАЗДЕЛ 2 — ПАЗАРЕН КОНТЕКСТ БЪЛГАРИЯ 2026

### 2.1 Еврозона — преходен период

От **1 януари 2026 г.** България е официален член на еврозоната.
Фиксиран обменен курс: **1 EUR = 1.95583 BGN**.

Важни последствия за продукта:

| Аспект | Ефект върху таблицата |
|---|---|
| Всички заплати се изплащат в EUR | Потребителят въвежда директно в EUR |
| Цените в магазините са в EUR | Kaufland/Lidl листовките са вече в EUR |
| Психологическо усещане за по-скъпо | Реалните суми са идентични — само мащабът изглежда по-малък |
| Данъчни ставки непроменени | Осигуровки и ДДФЛ остават 10% + ~13.78% |

### 2.2 Заплати в България — официални данни (2026)

Всички суми са **нетни (след данъци и осигуровки)** — именно това
потребителят получава на ръка и въвежда в таблицата.

| Показател | Брутно | Нетно | Забележка |
|---|---|---|---|
| **Минимална заплата** | €620.20 | ~€476 | От 01.01.2026; найниска в ЕС |
| **Медианна заплата** | ~€950 | ~€730 | Оценка; 50% от работещите под тази сума |
| **Средна заплата** | €1,369 | ~€1,062 | НСИ Q4 2025; +11% г/г |
| **IT сектор (средна)** | ~€3,000 | ~€2,250 | Силно влияние върху средното |
| **Обществен сектор** | ~€900 | ~€700 | Учители, здравеопазване |

> **Ключово за алгоритъма:** Таблицата работи с **нетната заплата**.
> При онбординг потребителят въвежда сумата, която реално получава —
> не брутото. Инструкцията в Setup листа изрично пояснява това.

**Как се изчислява нетото от брутото (справка):**
- ДДФЛ (данък): 10% върху данъчната основа
- Осигуровки (служител): ~13.78% (здравни + социални)
- Нетно ≈ Брутно × 0.762 (приближение за повечето случаи)

### 2.6 Регионална диференциация

Алгоритъмът на таблицата **не прави автоматична регионална корекция** в

## РАЗДЕЛ 3 — КЛАСОВЕ ДОХОДИ ЗА БЪЛГАРИЯ

### 3.1 Логика на класификация

Таблицата автоматично определя класа на потребителя веднага щом той
въведе **нетния си месечен доход** в Setup листа.

**Правило:** Класът се определя от **нетния доход** (сумата на ръка),
**не** от брутото. Това е съзнателно решение — потребителят не трябва
да изчислява данъци.

Алгоритъмът:
```
АКО доход < 650        → клас НИСЪК
АКО доход < 1 100      → клас НИСЪК СРЕДЕН
АКО доход < 1 800      → клас СРЕДЕН
АКО доход < 3 000      → клас ВИСОК СРЕДЕН
ИНАЧЕ                  → клас ВИСОК
```

---

### 3.2 Прагове на класовете (нетен доход/месец, EUR)

| Клас | Минимум | Максимум | % работещи в БГ | Типичен профил |
|---|---|---|---|---|
| **НИСЪК** | €0 | €649 | ~25% | Минимална заплата, регионални работници, part-time |
| **НИСЪК СРЕДЕН** | €650 | €1,099 | ~35% | Търговия, производство, малки градове, млади специалисти |
| **СРЕДЕН** | €1,100 | €1,799 | ~25% | Специалисти в София, учители, здравеопазване |
| **ВИСОК СРЕДЕН** | €1,800 | €2,999 | ~12% | IT, мениджъри, опитни специалисти |
| **ВИСОК** | €3,000 | ∞ | ~3% | Senior IT, директори, предприемачи |

---

### 3.3 Изчисление на дискреционния бюджет

Преди прилагане на процентите, алгоритъмът изчислява **дискреционния
бюджет** — парите, останали след всички фиксирани разходи:

```
Дискреционен бюджет = Нетен доход
                    − Наем/Ипотека
                    − Комунални
                    − Застраховки
                    − Вноски по кредити
                    − Спестовна цел
```

Процентите в таблица 3.4 се прилагат **върху дискреционния бюджет**,
не върху целия доход.

> **Граничен случай:** Ако дискреционният бюджет е ≤ €0 (наемът
> изяжда всичко), таблицата показва предупреждение и предлага
> преразглеждане на фиксираните разходи. Спестовната цел автоматично
> се намалява до €0 — но никога не се задава отрицателна.

### 3.6 Корекция при непропорционален наем

В България наемът в София може да изяде 50–70% от доход на НИСЪК клас.
В такъв случай алгоритъмът прилага следната корекция:

```
АКО (Наем / Нетен доход) > 40%:
    → Предупреждение: "Наемът ти е висок спрямо дохода"
    → Спестовна цел автоматично = €0
    → Процентите на ДИСКРЕЦИОННИ категории се намаляват с 30%
    → Разликата се добавя към ЗАЩИТЕНИ категории (Хранителни стоки)
```

## РАЗДЕЛ 5 — ОНБОРДИНГ

### 5.1 Цел и принципи

Онбордингът трябва да отговаря на три изисквания:

| Изискване | Целева стойност |
|---|---|
| Време за попълване | ≤ 3 минути |
| Брой задължителни полета | 5 (минимален жизнеспособен вход) |
| Резултат след попълване | Генериран бюджет + попълнен календар |

**Принципи:**
- Питаме само за това, което **не можем да изчислим** сами
- Предлагаме разумни стойности по подразбиране за всяко поле
- Не плашим с финансова терминология — прост разговорен език
- Потребителят може да промени всичко след генерирането

---

### 5.2 Структура на Setup листа

Setup листът е единственото място, което потребителят попълва ръчно.
Организиран е в **3 блока**.

```
Колона A  — Наименование на полето (само за четене)
Колона B  — Стойност (потребителят попълва тук)
Колона C  — Помощен текст / пример (само за четене, светлосив)
```

| 6 | Комунални услуги (€) | 130 | *Ток, вода, парно, интернет — средно за месеца* |
| 7 | Застраховки (€) | 0 | *Живот, имущество, кола (задължителна) и др.* |
| 8 | Вноски по кредити (€) | 0 | *Потребителски кредит, автокредит, студентски заем* |

**Тези 5 полета са достатъчни** за генериране на базов бюджет.

## РАЗДЕЛ 6 — РАЗПРЕДЕЛЕНИЕ ПО ДНИ

### 6.1 Цел на разпределението

Calendar листът показва за всеки ден от месеца **едно число** —
колко е планирано да се похарчи. Това число е сборът от всички
категории, разпределени към този ден.

Алгоритъмът трябва да отговори на въпроса:
> *„Кой лев от кой ден идва?"*

---

### 6.2 Трите типа поведение

#### FIXED — фиксиран ден
Цялата месечна сума на категорията пада на **един конкретен ден**.

| Правило | Ден |
|---|---|
| Спестявания, Авариен фонд, Наем, ДЗО | Ден **1** |
| Комунални, Застраховки, Вноски по кредити | Ден **5** |
| Абонаменти (Netflix, gym и др.) | Ден **1** |

> Ако ден 1 или ден 5 е събота/неделя — сумата **остава на същия ден**.
> Наемът не се мести — плаща се на договорената дата.

---

#### SPREAD — равномерно разпределение
Месечната сума се разделя равно по **делничните дни** (пон–пет).

Формула:
```
Сума за 1 ден = Месечна сума ÷ Брой делнични дни в месеца
```

Категории: Хранителни стоки, Кафе, Градски транспорт, Медицина,
Лична хигиена, Кола/Гориво.

**Защо само делнични:**
В събота/неделя потребителят вероятно яде навън (→ Хранене навън,
clustered), а не пазарува в Kaufland. Делничното spread отразява
реалното поведение.

**Частично изключение — Хранителни стоки:**
Добавя се и събота сутринта като пазарен ден:
```
Брой дни за spread = Делнични дни + Брой съботи в месеца
```
Резултатът: малко по-ниски делнични суми, малко по-висока събота.

---

#### CLUSTERED — групирани дни
Месечната сума се разпределя върху **случайно избрани ~40% от дните**
на месеца. Изборът е детерминиран (фиксиран за всяка категория, не се
мени при всяко отваряне на таблицата).

Формула за избор на дни:
```
seed = сумарна ASCII стойност на името на категорията
избрани_дни = deterministicSample(всички_дни, брой = месечни_дни × 0.4, seed)
сума_за_ден = Месечна сума ÷ брой_избрани_дни
```

**Съботно-неделен приоритет за clustered категории:**
Ресторантите, развлеченията и пазаруването се случват предимно в
уикенда. Затова при избора на дни за clustered категории:
```
Тегло на събота и неделя = 2.5× спрямо делничен ден
```
Практически резултат: ~60% от clustered дните падат в уикенда.

---

Следните дати са официални неработни дни и се третират като **неделя**
при spread изчислението (не се разпределят spread категории):

| Великденски понеделник | Подвижен |
| 1 май | Ден на труда |
| 6 май | Гергьовден |
| 24 май | Ден на просветата |

### 6.4 Пример за реално разпределение

Потребител, СРЕДЕН клас, €1,400 нето, май 2026 (31 дни, 21 делнични):

| Категория | Тип | Месечна сума | Ден(ове) | Сума/ден |
|---|---|---|---|---|
| Спестявания | fixed | €140 | Ден 1 | €140 |
| Наем | fixed | €500 | Ден 1 | €500 |
| Комунални | fixed | €130 | Ден 5 | €130 |
| Абонаменти | fixed | €22 | Ден 1 | €22 |
| Хранителни стоки | spread | €162 | 21 дел. + 4 съб. | €6.48/ден |
| Кафе | spread | €35 | 21 делнични | €1.67/ден |
| Транспорт | spread | €28 | 21 делнични | €1.33/ден |
| Хранене навън | clustered | €112 | ~12 дни (уикенд фокус) | €9.33/ден |
| Развлечения | clustered | €63 | ~12 дни | €5.25/ден |
| Облекло | clustered | €56 | ~12 дни | €4.67/ден |

**Ден 1 (четвъртък):** €140 + €500 + €22 + €6.48 + €1.67 + €1.33 = **€671.48**
**Типичен делничен ден:** €6.48 + €1.67 + €1.33 = **€9.48**
**Типичен уикенд ден:** €9.33 + €5.25 = **€14.58** (без clustered облекло)

### 6.5 Calendar лист — структура на редовете

Всеки ред в Calendar листа = 1 ден от месеца.

| Колона | Съдържание | Тип |
|---|---|---|
| A | Номер на деня (1–31) | Число |
| B | Дата (фор. `ДД МММ`) | Текст |
| C | Ден от седмицата (`Пон`, `Вт`...) | Текст |
| D | Планирано (€) | Число, 2 дес. |
| E | Похарчено (€) | Число, 2 дес. |
| F | Остатък (€) | Формула: =D−E |
| G | Статус | Текст: `зелен/жълт/червен/бъдещ` |
| H | Бележки | Текст (свободен) |

**Правило за колона F (Остатък):**
```
АКО E = 0 И денят е в бъдещето → показва D (планираното)
АКО E = 0 И денят е минал     → показва 0
АКО E > 0                     → показва D − E (може да е отрицателно)
```

---

### 6.6 Статус на деня

| Статус | Условие | Цвят на реда |
|---|---|---|
| **бъдещ** | Днес < ден в реда И E = 0 | Светлосиво `#F8F9FA` |
| **зелен** | E ≤ D | Светлозелено `#D4EDDA` |
| **жълт** | D < E ≤ D + праг | Светложълто `#FFF3CD` |
| **червен** | E > D + праг | Светлочервено `#F8D7DA` |

**Праг за жълт статус** (динамичен, не фиксиран):
```
праг = МАКС(€2, МИН(D × 5%, €25))
```
Примери:
- Планирано €10 → праг €2 (минимум)
- Планирано €50 → праг €2.50
- Планирано €200 → праг €10
- Планирано €800 → праг €25 (максимум)

> Жълтият статус **не задейства** ребалансъра. Само червеният задейства.

---

### 6.7 Какво НЕ трябва да прави разпределението

| Забранено поведение | Причина |
|---|---|
| Отрицателна планирана сума за ден | Математически невъзможно — сигнал за грешка |
| Планирана сума €0 за всеки ден | Показва проблем в онбординга |
| Разпределение на СВЕЩЕНИ категории в Calendar | Те се изваждат преди изчислението |
| Промяна на разпределението при всяко отваряне | Детерминираният seed осигурява стабилност |
| Разпределение след края на месеца | Ако месецът е 30 дни — ден 31 не съществува |

## РАЗДЕЛ 7 — ПРЕРАЗПРЕДЕЛЕНИЕ (REBALANCER)

> **Важно за разбирането на целия раздел:**
> Ребалансърът **не мести пари** между банкови сметки. Парите вече са
> похарчени — потребителят е платил в реалния свят. Това, което
> ребалансърът прави, е да **намали планираните лимити за бъдещите дни**,
> така че дневното число, което потребителят вижда, да отразява реалното
> му положение.
>
> Накратко: *„Похарчи €30 вместо €20 днес → утре ще ти покажа €18 вместо
> €20, за да не изгориш бюджета за месеца."*

---

### 7.1 Роля в продукта

Ребалансърът е **единствената причина**, поради която тази таблица е
различна от всички останали. Без него MITA е поредната бюджетна таблица
с червени клетки. С него — тя е жива система, която се настройва сама.

Трите задачи на ребалансъра:

| # | Задача | Ефект върху потребителя |
|---|---|---|
| 1 | Откривам превишения автоматично | Не мисли, не броя, не коригирам |
| 2 | Намаля бъдещите дни пропорционално на превишението | Всеки ден виждам актуалното число |
| 3 | Документирам всяка промяна в одитен лог | Виждам откъде са взети парите |

---

### 7.2 Условия за задействане

Ребалансърът се стартира **веднага** при следните условия (всички
трябва да са изпълнени):

| # | Условие | Проверка |
|---|---|---|
| 1 | Нова транзакция е добавена в Transactions листа | `onEdit` trigger на колона D |
| 2 | Денят, за който е транзакцията, е със статус **ЧЕРВЕН** след добавянето | `Spent > Planned + Yellow Threshold` |
| 3 | Превишението е ≥ €1 | Игнорираме нищожни суми |
| 4 | Има поне **1 бъдещ ден** в текущия месец | Не работи на последния ден |
| 5 | Категорията на превишението **не е СВЕЩЕНА** | Ако похарчиш повече за „Наем" — това е друг проблем |

**Какво НЕ задейства ребалансъра:**
- Жълт статус (в рамките на прага) — това е буфер
- Редактиране на стара транзакция — обработва се ръчно
- Превишение на месечен общ лимит без да е превишен конкретен ден
- Корекция от потребителя на планирана сума в Calendar листа

---

### 7.3 Скрита структура от данни: DayPlan

За да работи ребалансърът коректно, Calendar листът **не е достатъчен** —
той показва само общи суми на ден. Ребалансърът има нужда да знае
**коя категория е колко** на всеки ден, за да вземе от правилния източник.

Затова таблицата поддържа **скрит лист `_DayPlan`** (подчертаното означава
скрит от потребителя).

| Колона | Съдържание | Пример |
|---|---|---|
| A | Ден (1–31) | 15 |
| B | Категория | Хранене навън |
| C | Приоритет (0–3) | 3 |
| D | Поведение | clustered |
| E | Първоначално планирано (€) | 9.33 |
| F | Текущо планирано (€) | 9.33 |
| G | Похарчено (€) | 0 |
| H | Последна промяна (timestamp) | 2026-05-15 14:22 |

**Инварианти:**
- `_DayPlan` се генерира **веднъж** при Initial Setup
- Calendar D (Planned/ден) = `SUM(_DayPlan.F WHERE Day = n)` — формула
- Calendar E (Spent/ден) = `SUM(_DayPlan.G WHERE Day = n)` — формула
- Ребалансърът модифицира **само колона F** в `_DayPlan`

---

### 7.4 Подбор на донори

Когато превишението се регистрира, ребалансърът изгражда **донорски пул** —
списък с бъдещи записи в `_DayPlan`, от които може да вземе пари.

**Критерии за включване в пула:**

```
WHERE _DayPlan.Day     > overspendDay
  AND _DayPlan.Priority > 0                         (не СВЕЩЕНА)
  AND _DayPlan.Category ≠ overspentCategory         (не от себе си)
  AND _DayPlan.F        > минимален_праг(Category)
```

**Минимални прагове по категория** (никога не падаме под тях):

| Категория | Минимум на ден |
|---|---|
| Хранителни стоки | €3.50 |
| Градски транспорт | €0.80 (единичен билет) |
| Медицина | €0 (може да е напълно изпразнена) |
| Всички ГЪВКАВИ | €0 |
| Всички ДИСКРЕЦИОННИ | €0 |

**Сортиране на донорския пул (многониво):**

```
ORDER BY Priority       DESC,  -- ДИСКРЕЦИОННА първа (3 → 1)
         SubPriority    DESC,  -- виж 7.5
         CurrentAmount  DESC,  -- най-голямата сума първа
         Day            DESC   -- най-далечният ден първи
```

Логиката: Първо изпразваме най-безопасните за пропускане категории
(пътувания, подаръци, пазаруване). После — гъвкавите (кафе, козметика).
В краен случай — защитените.

---

### 7.5 Sub-приоритет в рамките на едно ниво

### 7.6 Правила за „вземане" (take rules)

За всеки донор, при вземане се прилагат следните ограничения:

| Правило | Стойност |
|---|---|
| Максимум от един запис в `_DayPlan` | **50% от текущата сума** |
| Минимална единична транзакция за вземане | **€0.50** (игнорираме по-малки) |
| Никога под минималния праг на категорията | Виж 7.4 |
| Максимум 20 трансфера в един ребаланс | Защита срещу безкрайни цикли |
| Никога от същата категория на превишението | Не крадем от себе си |

**Формула за реалното вземане от един донор:**

```
потенциал_на_донора = donor.F × 0.5
максимум_по_праг    = donor.F − минимум(donor.Category)
действително_вземане = МИНИМУМ(
    нужната_сума,
    потенциал_на_донора,
    максимум_по_праг
)
```

---

### 7.7 Пълен алгоритъм — стъпка по стъпка

```
ВХОД:  overspendDay, overspentCategory, overspendAmount

1.  АКО overspendAmount < €1 → ИЗХОД (не си струва)
2.  АКО overspendDay е последният ден на месеца → ИЗХОД
3.  АКО overspentCategory е СВЕЩЕНА → ИЗХОД (logging only)

4.  Извлечи донорски пул (виж 7.4)
5.  Сортирай донорите (виж 7.4 + 7.5)

6.  remaining = overspendAmount
7.  transfers = []
8.  maxTransfers = 20

9.  FOR donor IN донорски_пул:
        АКО remaining ≤ €0.50 → STOP
        АКО LEN(transfers) ≥ maxTransfers → STOP

        take = изчисли_вземане(donor, remaining)  (виж 7.6)
        АКО take < €0.50 → CONTINUE

        donor.F -= take
        donor.H = now()
        remaining -= take
        transfers.push({donor, take})

10. АКО LEN(transfers) > 0:
        overspentDayPlan.F += SUM(transfers.take)  (кредитиране)
        overspentDayPlan.H = now()

11. ЛОГВАЙ всичко в Rebalance листа (виж 7.8)

12. ПРЕСМЕТНИ отново Calendar D и E за засегнатите дни
    (автоматично чрез формулите)

13. ПОКАЖИ toast: „Ребалансирано: €X от N дни"

14. АКО remaining > €0.50:
        overspentDay.Notes = „⚠ Недостиг €X не можа да бъде покрит"
        Denят остава ЧЕРВЕН
```

### 7.8 Механизъм „кредитиране обратно" (credit back)

Това е ключовата иновация, която отличава MITA от конкурентите.

**Проблем:** Ако просто намалим бъдещите дни, денят на превишението
остава завинаги червен в историята. Потребителят чувства „провал".

**Решение:** След успешен ребаланс, денят на превишението получава
**увеличена планирана сума** с размера на покритата сума. Практически:

| Момент | Planned | Spent | Status |
|---|---|---|---|
| Преди транзакцията | €20 | €15 | Зелен |
| Веднага след €20 за хранене | €20 | €35 | Червен |
| След ребаланса (взети €15) | **€35** | €35 | **Зелен** |

> Денят е отново зелен, защото планираното вече отразява реалното
> положение. Потребителят вижда зелено. Rebalance логът показва
> точната история за прозрачност.

**Частично покритие:** Ако ребалансърът покрие само €12 от €15:

| Момент | Planned | Spent | Status |
|---|---|---|---|
| След ребаланса | €32 | €35 | Червен (€3 непокрити) |

Денят остава червен, но с по-малка разлика. Бележка в колона H:
*„Недостиг €3 не можа да бъде покрит"*.

### 7.9 Логване в Rebalance листа

Всяко единично вземане се записва в Rebalance листа като отделен ред.

| Колона | Съдържание | Пример |
|---|---|---|
| A | Timestamp | 2026-05-15 14:22:11 |
| B | Ден на превишението | 15 |
| C | Категория на превишението | Хранене навън |
| D | Размер на превишението | €15.00 |
| E | Ден донор | 22 |
| F | Категория донор | Пътувания |
| G | Взета сума | €9.33 |
| H | Остатък непокрит | €5.67 |
| I | Бележка | — |

**Обобщение на ребаланса (един ред накрая):**

|---|---|---|---|---|---|---|---|---|
| 2026-05-15 14:22:11 | 15 | — | — | — | **ОБОБЩЕНИЕ** | €15.00 | €0.00 | 2 трансфера, покрити €15 |

### 7.11 Гранични случаи

#### Случай А: Недостатъчен бъдещ бюджет

```
Условие:   Превишение €50, но всички бъдещи дискреционни = €30
Поведение: Взима се €30 (максимум възможно)
           Денят остава червен с €20 недостиг
           Бележка: „⚠ Недостиг €20 не може да бъде покрит"
           Rebalance лог: отделни трансфери + обобщителен ред
```

#### Случай Б: Последователни превишения

```
Условие:   Ден 10 превишение € 5 → ребаланс успешен
           Ден 11 превишение €10 → ребаланс (с намалени бъдещи дни)
           Ден 12 превишение €8  → ребаланс с още по-малко ресурс
Поведение: Всеки ребаланс работи с актуалното състояние на `_DayPlan`
           Не прави „супер-ребаланс" или глобално преизчисление
           Ако месецът е в криза → червените дни се натрупват
```

#### Случай В: Превишение на последния ден

```
Условие:  Ден 31 превишение €20, няма бъдещи дни
Поведение: Ребалансърът НЕ се стартира (условие 7.2 #4)
          Денят остава червен завинаги
          Бележка: „Край на месеца — без възможност за ребаланс"
```

#### Случай Г: Превишение в СВЕЩЕНА категория

```
Условие:   Потребителят плати €50 повече за наем
Поведение: Ребалансърът НЕ се стартира
           Показва се съобщение: „Превишение в свещена категория.
           Провери Setup листа и обнови бюджета си."
           Логва се в Rebalance листа като тип „SACRED_OVERFLOW" за одит
```

#### Случай Д: Транзакция със задна дата

```
Условие:   На 15-и потребителят вписва транзакция за 10-и
Поведение: Ребалансърът работи нормално, но търси донори
           само между 15-и и края на месеца (не между 10-и и 15-и)
           Дни 10–14 са вече в миналото и не могат да бъдат променени
```

#### Случай Е: Корекция на стара транзакция

```
Условие:   Потребителят трие или променя транзакция от 10-и на 15-и
Поведение: Ребалансърът НЕ се стартира автоматично (условие 7.2 #1)
           Потребителят трябва да натисне „Преизчисли всички"
           от менюто, което прави пълно преизчисление на Calendar
           без да задейства rebalancer (виж 7.13)
```

### 7.12 Ръчна намеса и отмяна

#### Бутон „Отмени последния ребаланс"

Потребителят може да отмени последния ребаланс чрез менюто.
Действието:

1. Намира последния блок записи в Rebalance листа (група по timestamp)
2. Връща всеки донор с неговата взета сума
3. Намаля overspentDay.F обратно
4. Добавя обобщителен запис: *„REVERTED by user"*

**Ограничение:** Може да отмени **само последния** ребаланс.
Не поддържаме дълбоко undo в v1.

#### Бутон „Изключи авто-ребаланс"

Advanced feature, скрит в настройки. Когато е изключен:
- Ребалансърът не се стартира автоматично
- Червените дни остават червени
- Потребителят може ръчно да стартира: „Ребалансирай сега"

Полезно за хора, които искат да видят реалните си червени дни,
без автоматично „изглаждане".

---

### 7.13 Функция „Преизчисли всички"

Това е безопасна функция за reset. Действието:

```
1. НЕ задейства ребалансъра
2. Чете всички транзакции от Transactions листа
3. Нулира _DayPlan.G за всички редове
4. Обработва отново всяка транзакция:
   - Намира съответната категория и ден
   - Увеличава _DayPlan.G
5. Преизчислява статусите по новите данни
6. НЕ пипа _DayPlan.F (запазва ефектите от минали ребаланси)
```

Използва се след ръчна корекция на транзакция.

### 7.14 Забранени поведения

| Забранено | Защо |
|---|---|
| Вземане от СВЕЩЕНА категория | Нарушение на принципа |
| Вземане от минал ден | Времето не се връща |
| Вземане от същата категория на превишението | Абсурдно — крадеш от себе си |
| Ребаланс без запис в Rebalance листа | Пълна прозрачност винаги |

| Мълчаливо скриване на недостиг | Винаги показва бележка ако не е покрито |
| Рекурсивен ребаланс (ребаланс на ребаланс) | Единичен pass, без loop |
| Ребаланс на транзакции под €1 | Шум, не сигнал |

## РАЗДЕЛ 8 — ЦЕЛИ (GOALS)

### 8.1 Философия

Бюджетирането без цел е задушаващо. MITA третира целите като
**цел на целия продукт**, а не като допълнителна функция.
Всеки лев, който не е похарчен за разход, е лев, който движи
потребителя към конкретен резултат.

Три принципа:

| Принцип | Значение |
|---|---|
| 1. Целите са свещени | Ребалансърът **никога** не ги докосва |
| 2. Целите са видими | Прогресът се показва на всеки лист, не само на Goals |
| 3. Целите се празнуват | Достигането на цел задейства визуална обратна връзка |

---

### 8.2 Три типа цели

#### Тип 1 — Месечна спестовна цел (основна)

Това е **единствената задължителна** цел в таблицата. Представлява
сумата, която потребителят иска да задели **всеки месец**.

| Характеристика | Стойност |
|---|---|
| Приоритет | СВЕЩЕНА (0) |
| Поведение | fixed (ден 1) |
| Източник | Setup лист, поле „Месечна спестовна цел" |
| Авто-изчисление | Да (виж Раздел 5.4) |
| Може ли да бъде €0 | Да (за НИСЪК клас) |
| Участва в Calendar | Не — изважда се от дохода преди разпределението |

**Поведение на 1-во число:**
Таблицата предполага, че потребителят е прехвърлил сумата към спестовна
сметка на 1-во число. Потребителят потвърждава това с чек-бокс в
Goals листа („Прехвърлих спестяването този месец").

---

#### Тип 2 — Авариен фонд (дългосрочна цел)

Авариен фонд е **кумулативна цел** — потребителят натрупва до определена
обща сума (обикновено 3–6 месечни заплати).

| Характеристика | Стойност |
|---|---|
| Тип | Кумулативна |
| Препоръчителен размер | 3 × нетен месечен доход |
| Месечна вноска | Потребителят избира |
| Приоритет на вноската | СВЕЩЕНА (0) |
| Статус при достигане | Замразява се — вноската става €0 |

**Когато фондът е пълен:** таблицата предлага:
- Опция А: Преместване на месечната вноска към други цели
- Опция Б: Увеличаване на месечната спестовна цел
- Опция В: Добавяне на нова дългосрочна цел

---

#### Тип 3 — Персонализирани цели (незадължителни)

Потребителят може да създаде **до 5 персонализирани цели** (v1 ограничение)
с име, целева сума и целева дата.

Примери:
- „Почивка Гърция 2026" — €800 до юли
- „Нов лаптоп" — €1,200 до декември
- „Сватбен подарък за брат ми" — €150 до септември
- „Първа вноска за апартамент" — €15,000 до 2028

Всяка цел има три режима на финансиране:

| Режим | Описание |
|---|---|
| **Месечна вноска** | Фиксирана сума всеки месец, включена в СВЕЩЕНИ разходи |
| **Единични преводи** | Потребителят вписва ръчно всеки превод към целта |
| **Остатък** | Всичко, което остане в края на месеца, отива в целта |

---

### 8.3 Goals лист — структура

Нов лист, добавен към таблицата (не беше в оригиналния v0 списък).

| Колона | Съдържание | Тип |
|---|---|---|
| A | Име на целта | Текст |
| B | Тип | Enum: `Месечна / Авариен / Персонализирана` |
| C | Целева сума (€) | Число |
| D | Натрупано до момента (€) | Число |
| E | Оставащо (€) | Формула: `=C−D` |
| F | Прогрес (%) | Формула: `=D/C` |
| G | Визуален прогрес бар | Формула с REPT |
| H | Месечна вноска (€) | Число |
| I | Целева дата | Дата (незадължително) |
| J | Прогнозна дата на достигане | Формула |
| K | Статус | Enum: `Активна / Замразена / Завършена / Просрочена` |
| L | Последна вноска (дата) | Дата |
| M | Бележка | Текст |

#### Формула за визуалния прогрес бар (колона G)

```
= REPT("█", ROUND(F*20, 0)) & REPT("░", 20 - ROUND(F*20, 0))
```

Примери:
- 0% → `░░░░░░░░░░░░░░░░░░░░`
- 25% → `█████░░░░░░░░░░░░░░░`
- 60% → `████████████░░░░░░░░`
- 100% → `████████████████████`

#### Формула за прогнозна дата (колона J)

```
АКО месечна_вноска > 0:
    оставащи_месеци = ROUNDUP(оставащо / месечна_вноска, 0)
    прогнозна_дата = EDATE(днес, оставащи_месеци)
ИНАЧЕ:
    „—"
```

### 8.4 Прогрес-тракинг — как се попълва колона D

Има **три механизма** за попълване на „Натрупано до момента":

#### Механизъм 1 — Автоматичен (за месечната спестовна цел)

На 1-во число, когато потребителят отбележи „Прехвърлих":
```
Goals.D[Месечна] += Setup.МесечнаСпестовнаЦел
Records.append(дата, сума, „авто-месечна")
```

#### Механизъм 2 — Ръчен превод

Потребителят натиска бутона „Добави превод към цел" от менюто MITA.
Диалогов прозорец пита:
- Коя цел?
- Колко?
- Откъде идват парите? (заплата / друг източник)
- Бележка

Създава се запис в скрития лист `_GoalLedger`.

#### Механизъм 3 — Остатъчен преливник

На последния ден на месеца (ден 30 или 31), ако:
- Няма нито един червен ден
- Има положителен общ остатък в Calendar листа
- Поне една цел има режим „Остатък"

Тогава таблицата предлага:
```
„Имаш €45 неизползван бюджет този месец!
 Искаш ли да ги добавиш към „Почивка Гърция 2026"?"
```

При ДА → парите автоматично се прехвърлят към целта.

### 8.5 Скрит лист `_GoalLedger`

Пълна история на всички движения по цели. Използва се за одит и
възстановяване на колона D.

| Колона | Съдържание | Пример |
|---|---|---|
| A | Дата | 2026-05-01 |
| B | Цел | Почивка Гърция 2026 |
| C | Сума (+/−) | +80.00 |
| D | Тип | месечна / ръчна / остатък / теглене |
| E | Бележка | — |
| F | Баланс след трансакцията | 320.00 |

**Инвариант:** `Goals.D[X] = SUM(_GoalLedger.C WHERE _GoalLedger.B = X)`
— тоест колона D в Goals листа винаги е формула, не статично число.

### 8.6 Визуализация на прогреса извън Goals листа

Целите трябва да са **видими постоянно**, не само когато потребителят
отвори Goals листа.

| Лист | Къде се показва | Какво се показва |
|---|---|---|
| Setup | Под полето „Месечна спестовна цел" | „Натрупано: €XXX / €YYY (зз%)" |
| Monthly | Секция „Цели" (нов блок) | Списък с всички активни цели + прогрес бар |
| Calendar | Заглавна лента над дните | „Цел: Почивка Гърция | ███████░░ 65%" |
| Transactions | — | Не се показва (за да не пречи) |

### 8.7 Връзка с ребалансъра

**Абсолютно правило:** Ребалансърът **никога** не взима пари от цели.

В `_DayPlan` цели имат приоритет 0 (СВЕЩЕНА) и са изключени от донорския
пул по правилото в 7.4.

**Допълнителна защита:** Дори ако потребителят ръчно добави цел с
приоритет ≠ 0 в Categories, скриптът при следващия ребаланс проверява
името на категорията срещу списъка в Goals листа и автоматично връща
приоритета на 0.

### 8.8 Месечен workflow на целите

#### Начало на месеца (ден 1)

```
1. Показва се диалогов прозорец „Нов месец — твоите цели":
   ┌──────────────────────────────────────┐
   │ Месечна спестовна цел: €140          │
   │ ☐ Прехвърлих спестяването            │
   │                                      │
   │ Авариен фонд: €50                    │
   │ ☐ Прехвърлих                         │
   │                                      │
   │ Почивка Гърция 2026: €80             │
   │ ☐ Прехвърлих                         │
   │                                      │
   │ [Потвърди]  [Направи го по-късно]    │
   └──────────────────────────────────────┘

2. За всеки отметнат:
   - Goals.D се увеличава
   - _GoalLedger получава запис
   - Notification при достигане на цел

3. За неотметнатите:
   - Поява на напомняне след 3 дни
   - След 7 дни — червено предупреждение в Monthly листа
```

#### Край на месеца (ден 30 или 31)

```
1. Проверка за неизразходван бюджет (виж Механизъм 3)
2. Архивиране на месечния прогрес
3. Обобщение на постигнатото:
   „Този месец спести €140 + €50 + €80 = €270"
4. Показване на обобщителен диалог:
   „Този месец: €270 към целите ти
    Общо спестени от началото: €1,240"
```

## РАЗДЕЛ 9А — РЕАЛНИ ЦЕНИ: АРХИТЕКТУРА И `_PRICES` ЛИСТ

### 9А.1 Защо не са достатъчни статични оценки

Всяка бюджетна таблица, която съдържа хардкодирани числа (например
„Хранителни стоки = €160/месец"), остарява за 3–6 месеца заради
инфлацията и сезонните колебания. MITA решава това с **жива ценова
база**: скрит лист `_Prices`, от който Categories листът изчислява
сумите в реално време.

Предимства:

| Предимство | Ефект |
|---|---|
| Числата се обновяват централно | Промяна в `_Prices` → всички категории актуализирани |
| Клиентът не вижда изчисленията | `_Prices` е скрит и защитен |
| Различни суми по клас | Един лист → 5 различни резултата |
| Одитируемост | Всяка цена има дата и источник |

---

### 9А.2 Структура на `_Prices` листа

Листът съдържа ~100 реда — по един за всеки трекван продукт или услуга.

| Кол. | Наименование | Тип | Пример |
|---|---|---|---|
| A | Код на категория | Текст | `groceries` |
| B | Под-категория | Текст | `dairy` |
| C | Продукт (БГ) | Текст | `Кисело мляко 400г` |
| D | Единица | Текст | `1 бр.` |
| E | Мин. цена (€) | Число | 0.75 |
| F | Средна цена (€) | Число | 0.88 |
| G | Макс. цена (€) | Число | 1.10 |
| H | К-во/месец — НИСЪК | Число | 4 |
| I | К-во/месец — НИС.СР. | Число | 5 |
| J | К-во/месец — СРЕДЕН | Число | 6 |
| K | К-во/месец — ВИС.СР. | Число | 7 |
| L | К-во/месец — ВИСОК | Число | 8 |
| M | Включен в кошница | TRUE/FALSE | TRUE |

> **Заглавен ред:** Ред 1 съдържа имената на колоните.
> Данните започват от ред 2.
> Общо: ~100 продуктови реда + секционни заглавия (само за четимост).

### 9А.4 Как Categories листът чете от `_Prices`

Всяка категория в Categories листа има **динамична формула** в колона B
(Месечен бюджет), вместо статично число.

#### Пример: Хранителни стоки

```
= SUMPRODUCT(
    FILTER(_Prices!F:F, (_Prices!A:A="groceries") * (_Prices!M:M=TRUE)),
    CHOOSE(
      MATCH(Setup!_IncomeClass, {"НИСЪК","НИС.СР.","СРЕДЕН","ВИС.СР.","ВИСОК"}, 0),
      FILTER(_Prices!H:H, (_Prices!A:A="groceries") * (_Prices!M:M=TRUE)),
      FILTER(_Prices!I:I, (_Prices!A:A="groceries") * (_Prices!M:M=TRUE)),
      FILTER(_Prices!J:J, (_Prices!A:A="groceries") * (_Prices!M:M=TRUE)),
      FILTER(_Prices!K:K, (_Prices!A:A="groceries") * (_Prices!M:M=TRUE)),
      FILTER(_Prices!L:L, (_Prices!A:A="groceries") * (_Prices!M:M=TRUE))
    )
  ) * Setup!_MembersCount
```

**Логика:** За всеки продукт в `groceries` → взима средната цена (кол. F)
и количеството за съответния клас (кол. H–L) → умножава → сумира →
умножава по брой членове.

#### Пример: Транспорт (фиксирана услуга, не продукт)

```
= IFERROR(
    FILTER(_Prices!F:F,
      (_Prices!A:A="transport") *
      (_Prices!C:C=Setup!_CityTransportKey)
    ),
    28
  )
```

`Setup!_CityTransportKey` е наименован диапазон, зависещ от избрания
град в Setup: `"sofia_monthly_card"`, `"plovdiv_monthly_card"`, и т.н.

#### Пример: Развлечения (брой посещения × цена)

```
= SUMPRODUCT(
    FILTER(_Prices!F:F, _Prices!A:A="entertainment"),
    CHOOSE(income_class_index,
      FILTER(_Prices!H:H, _Prices!A:A="entertainment"),
      ...
    )
  )
```

| Наименование | Диапазон | Описание |
|---|---|---|

### 9А.6 Обновяване на `_Prices` — техническа процедура

Когато ценовата база се обновява (нови данни от Kaufland, NSI и т.н.),
процесът е:

```
1. Отваряме _Prices листа (временно го правим видим в Apps Script)
2. Намираме реда по колона C (продукт) или A+C комбинация
3. Обновяваме колони E, F, G (мин/ср/макс цена)
4. Обновяваме колона O (дата)
5. Обновяваме колона N (источник)
6. Скриваме листа отново
7. Всички формули в Categories се преизчисляват автоматично
```

> **Версиониране:** Колона O (дата) позволява да се провери кога е
> обновена всяка цена. В Monthly листа се показва:
> *„Ценова база: 15 Апр 2026 — обнови ако е по-стара от 30 дни"*

### 9В.4 Версиониране на `_Prices`

Всяка версия на ценовата база е обозначена в **ред 1, колона R** на
`_Prices` листа:

```
_Prices!R1 = "v1.3 | 2026-04-16 | Седмично обновяване"
```

Формат: `vMAJOR.MINOR | ГГГГ-ММ-ДД | Тип обновяване`

**В Monthly листа** се показва предупреждение ако данните са стари:

```
АКО ДНЕС() − DATEVALUE(_Prices!O_LastUpdatedMin) > 60 ДЕНА:
    → Жълта клетка: „⚠ Ценовата база е на [X] дни.
       Препоръчваме обновяване."
АКО > 90 ДЕНА:
    → Червена клетка: „⚠ Ценовата база е стара!
       Вероятно числата са неточни."
```

`_Prices!O_LastUpdatedMin` = `MIN(_Prices!O:O)` — най-старата дата в листа.

### 9В.6 Индикатори за качество на данните

Всеки ред в `_Prices` има имплицитно ниво на доверие, базирано
на источника:

| Ниво | Тип источник | Примери | Маркер |
|---|---|---|---|
| ⭐⭐⭐ Висок | Официален публичен тариф | EVN, sofiatraffic.bg, spotify.com | `official` |
| ⭐⭐ Среден | Агрегатор с реални данни | fuelo.net, NSI, foody.bg | `aggregated` |
| ⭐ Нисък | Ръчна извадка / оценка | Посещение на магазин, средна от 3 сайта | `estimated` |

В `_Prices` колона N (Источник) съдържа маркера плюс URL.

**Правило:** Ако даден ред има маркер `estimated` и е по-стар от
60 дни → при обновяване се третира с приоритет.

### 9В.7 Процедура при липсващ или недостъпен источник

```
СИТУАЦИЯ: Источникът е недостъпен (сайтът е офлайн, 403, и т.н.)

ДЕЙСТВИЕ:
  1. Оставяме старата стойност (не я изтриваме)
  2. Добавяме суфикс в колона N: "[НЕДОСТЪПЕН - дата]"
  3. Маркираме ред в оранжево в _Prices
  4. Търсим алтернативен источник (напр. NSI вместо direktен магазин)
  5. Ако не намерим → оставяме старата стойност + маркер
  6. При следващо обновяване опитваме отново

НИКОГА не поставяме нулева или отрицателна цена при липса на данни.
```

### 9В.8 Процедура при аномалия в цената

```
СИТУАЦИЯ: Открита цена е >30% различна от предишната стойност

ПРИМЕРИ:
  - Пилешко гърди: беше €5.50, сега виждаме €3.20 (−42%)
  - Дизел: беше €1.77, сега виждаме €2.50 (+41%)

ДЕЙСТВИЕ:
  1. Проверяваме втори источник
  2. АКО потвърдено → обновяваме + добавяме в changelog с "⚠ АНОМАЛИЯ"
  3. АКО не потвърдено → оставяме старата стойност + бележка "Неверифицирано"
  4. При следваща сесия проверяваме отново

ПРАГОВЕ ЗА АНОМАЛИЯ:
  Храна, облекло, услуги: >25% промяна
  Гориво: >15% промяна (по-волатилно)
  Абонаменти (фиксирани): >0% = аномалия (проверяваме всяка промяна)
```

```javascript
function ensureOnEditTrigger_() {

    t.getHandlerFunction() === 'onEditInstallable' &&
    t.getEventType() === ScriptApp.EventType.ON_EDIT
  );
  if (!exists) {
    ScriptApp.newTrigger('onEditInstallable')
      .forSpreadsheet(SpreadsheetApp.getActive())
      .onEdit()
      .create();
  }
}
```

```javascript
function getCategoryBudget(catCode, data) {
  const prices = ss.getSheetByName(SHEETS.PRICES);
  const rows   = prices.getDataRange().getValues();

  // Намери тиерния индекс (0=LOW … 4=HIGH)
  const tierIdx = ['НИСЪК','НИС.СР.','СРЕДЕН','ВИС.СР.','ВИСОК']
                    .indexOf(data.incomeClass);          // → 0..4

  // Колони H..L = qty за 5 тиера (индекси 7..11)
  // Колона F = avg price (индекс 5)
  // Колона M = included flag (индекс 12)
  // Колона A = catCode (индекс 0)

  let total = 0;
  for (let i = 1; i < rows.length; i++) {
    const row = rows[i];
    if (row[0] !== catCode) continue;
    if (row[12] !== true)   continue;   // included = FALSE → пропусни

    const avgPrice = row[5];            // Avg €
    const qty      = row[7 + tierIdx];  // Qty за текущия тиер
    total += avgPrice * qty;
  }

  return roundEur(total * data.members);
}
```

```javascript
function buildMonthly(data) {
```

```javascript
function getCategoryBudget_(catCode, incomeClass, members) {
  const priceSheet = getSheet(SHEETS.PRICES);
  const data = priceSheet.getDataRange().getValues();

  const colIdx = { LOW: 7, 'НИС.СР.': 8, СРЕДЕН: 9, 'ВИС.СР.': 10, ВИСОК: 11 };
  const qtyCol = colIdx[incomeClass] ?? 9;

  let total = 0;
  for (let i = 1; i < data.length; i++) {
    const row = data[i];
    if (row[0] !== catCode) continue;    // A: category code
    if (row[12] !== true) continue;      // M: included flag
    const avgPrice = row[5];             // F: avg €
    const qty      = row[qtyCol];        // H-L: quantity per tier
    total += avgPrice * qty;
  }
  // Умножи по брой членове само за категории свързани с хора
  const perPersonCats = ['groceries','coffee','personal_care','medical'];
  if (perPersonCats.includes(catCode)) total *= members;

  return roundEur(total);
}
```

### 11Б.3 buildCalendar_() — Calendar.gs

**Роля:** Генерира 31-те реда на Calendar sheet с планирани суми по дни.

**Сигнатура:** `function buildCalendar_(yearMonth: string, incomeClass: string, members: number): void`

**Алгоритъм:**

```
1. Прочети Categories → списък {code, monthlyBudget, priority, behavior}
2. days = getDaysInMonth(yearMonth)
3. За всяка категория изчисли dailyAmounts[] с размер days
   - FIXED: само ден 1 получава пълната сума; останалите = 0
   - SPREAD: monthlyBudget / days (равномерно)
   - CLUSTERED: сумата се разпределя само в събота/неделя/празници
     seed = hash(yearMonth + catCode)
     rng  = deterministicRng(seed)
     clusteredDays = дни, където isWeekend или isHoliday
     разпредели пропорционално с малко шум от rng
4. plannedPerDay[d] = SUM(dailyAmounts[d] за всички категории)
5. Запиши Calendar:
   Row per day: [dayN, date, dayOfWeek, planned, 0, planned, 'СИВО', '']
6. Форматирай: header row navy/yellow, данни с alternating rows
7. Замрази ред 1
```

**Детайл за CLUSTERED разпределение:**
```javascript
function distributeClusteredDays_(monthlyBudget, yearMonth, catCode) {
  const days = getDaysInMonth(yearMonth);
  const [year, mon] = yearMonth.split('-').map(Number);
  const amounts = new Array(days).fill(0);

  const clusterDays = [];
  for (let d = 1; d <= days; d++) {
    const dt = new Date(year, mon - 1, d);
    if (isWeekend(dt) || isHoliday(dt)) clusterDays.push(d);
  }
  if (clusterDays.length === 0) {
    // Няма уикенди — fallback към spread
    return new Array(days).fill(roundEur(monthlyBudget / days));
  }

  const seed = simpleHash_(yearMonth + catCode);
  const rng  = deterministicRng(seed);

  // Генерирай тегла с малко случайност
  const weights = clusterDays.map(() => 0.8 + rng() * 0.4);
  const wSum = weights.reduce((a, b) => a + b, 0);

  clusterDays.forEach((d, i) => {
    amounts[d - 1] = roundEur(monthlyBudget * weights[i] / wSum);
  });

  // Коригирай закръгляне
  const diff = roundEur(monthlyBudget - amounts.reduce((a, b) => a + b, 0));
  if (diff !== 0) amounts[clusterDays[0] - 1] = roundEur(amounts[clusterDays[0] - 1] + diff);

  return amounts;
}
```

### 11Б.4 buildDayPlan_() — DayPlan.gs

**Роля:** Попълва скрития `_DayPlan` лист — per-категория per-ден tracking таблица.

**Сигнатура:** `function buildDayPlan_(yearMonth: string): void`

**Структура на `_DayPlan` (колони A–L):**

| A | B | C | D | E | F | G | H | I | J | K | L |
|---|---|---|---|---|---|---|---|---|---|---|---|
| Day# | CatCode | Priority | Behavior | Planned€ | Remaining€ | Locked | MinFloor€ | MaxTake% | DonorRank | LastModTS | Notes |

- Редове: `days × categories` (напр. 31 × 24 = 744 реда)
- `F (Remaining)` = `D (Planned)` при старт; намалява при транзакции и rebalance
- `G (Locked)` = TRUE ако денят е минал и е финализиран
- `H (MinFloor)` = минималната сума под която донорът не може да слезе (€1.00 за DISCRETIONARY, €0 за FLEXIBLE)
- `I (MaxTake%)` = 0.50 (50% правило)
- `J (DonorRank)` = 1 (DISCRETIONARY дава пръв) … 4 (PROTECTED дава последен)

```javascript
function buildDayPlan_(yearMonth) {
  const cats   = getCategoriesData_();   // чете Categories sheet
  const days   = getDaysInMonth(yearMonth);
  const sheet  = getOrCreateSheet(SHEETS.DAY_PLAN);
  sheet.clearContents();

  const header = ['Day#','CatCode','Priority','Behavior',
                  'Planned€','Remaining€','Locked','MinFloor€',
                  'MaxTake%','DonorRank','LastModTS','Notes'];
  const rows = [header];

  for (let d = 1; d <= days; d++) {
    for (const cat of cats) {
      const planned = cat.dailyAmounts[d - 1] || 0;
      rows.push([
        d,
        cat.code,
        cat.priority,
        cat.behavior,
        planned,
        planned,        // Remaining = Planned при старт
        false,          // Locked
        cat.minFloor,
        0.50,           // MaxTake 50%
        cat.donorRank,
        '',
        ''
      ]);
    }
  }

  sheet.getRange(1, 1, rows.length, 12).setValues(rows);
}
```

### 11Б.6 processTransactionRow() — Transactions.gs

**Роля:** Обработва един ред от Transactions при въвеждане/редактиране.

**Сигнатура:** `function processTransactionRow(sheet: Sheet, row: number): void`

**Алгоритъм:**

```
1. Прочети ред: [date, dayN, catCode, amount, note, processed]
2. АКО processed === 'OK' → ИЗХОД (вече обработен)
3. Валидирай: date, dayN (1-31), catCode (в Categories), amount > 0
4. АКО невалиден → маркирай с '#ERR' в F, покажи грешка → ИЗХОД
5. Намери редовете в _DayPlan за dayN + catCode
6. Обнови Remaining в _DayPlan: Remaining -= amount
7. Обнови Calendar!E[dayN] (Spent): += amount
8. Изчисли Calendar!F[dayN] (Remaining): = Planned - Spent
9. Изчисли статус:
   yellowThreshold = MAX(2, MIN(planned × 5%, 25))
   АКО spent <= planned → GREEN
   АКО spent <= planned + yellowThreshold → YELLOW
   АКО spent >  planned + yellowThreshold → RED → rebalanceFutureDays(dayN, catCode, overspend)
10. Оцвети ред dayN в Calendar (зелено/жълто/червено)
11. Запиши 'OK' в Transactions!F[row]
12. logChange_(...)
```

```javascript
function processTransactionRow(sheet, row) {
  const vals = sheet.getRange(row, 1, 1, 6).getValues()[0];
  const [date, dayN, catCode, amount, note, processed] = vals;

  if (processed === 'OK') return;

  // Валидация
  if (!dayN || dayN < 1 || dayN > 31 || !catCode || !amount || amount <= 0) {
    sheet.getRange(row, 6).setValue('#ERR');
    toast('⚠️ Невалидна транзакция на ред ' + row, 'MITA грешка');
    return;
  }

  const ss = SpreadsheetApp.getActive();

  // Обнови _DayPlan
  updateDayPlanRemaining_(dayN, catCode, amount);

  // Обнови Calendar
  const calSheet = getSheet(SHEETS.CALENDAR);
  const calRow   = dayN + 1; // ред 1 = header
  const planned  = calSheet.getRange(calRow, 4).getValue();
  const spent    = (calSheet.getRange(calRow, 5).getValue() || 0) + amount;
  const remaining = roundEur(planned - spent);

  calSheet.getRange(calRow, 5).setValue(spent);
  calSheet.getRange(calRow, 6).setValue(remaining);

  // Статус
  const yThresh = Math.max(2, Math.min(planned * 0.05, 25));
  let status, bgColor;
  if (spent <= planned) {
    status = STATUS.GREEN; bgColor = '#D9EAD3';
  } else if (spent <= planned + yThresh) {
    status = STATUS.YELLOW; bgColor = '#FFF2CC';
  } else {
    status = STATUS.RED; bgColor = '#F4CCCC';
    const overspend = roundEur(spent - planned - yThresh);
    rebalanceFutureDays(dayN, catCode, overspend);
  }

  calSheet.getRange(calRow, 7).setValue(status);
  calSheet.getRange(calRow, 1, 1, 8).setBackground(bgColor);

  sheet.getRange(row, 6).setValue('OK');
}
```

```javascript
function refreshCalendarRow_(dayN) {
  const calSheet  = getSheet(SHEETS.CALENDAR);
  const calRow    = dayN + 1;

  // Сумирай всички транзакции за деня
  const txSheet = getSheet(SHEETS.TRANSACTIONS);

  for (let i = 1; i < txData.length; i++) {

    }

  const remaining = roundEur(planned - spent);
  const yThresh   = Math.max(2, Math.min(planned * 0.05, 25));

  let status, bg;
  const today = new Date();
  const [yr, mo] = Props.get('BUDGET_MONTH').split('-').map(Number);
  const dayDate = new Date(yr, mo - 1, dayN);

  } else if (spent <= planned + yThresh) {
    status = STATUS.YELLOW; bg = '#FFF2CC';
  } else {
    status = STATUS.RED; bg = '#F4CCCC';
  }

  calSheet.getRange(calRow, 1, 1, 8).setBackground(bg);
```
