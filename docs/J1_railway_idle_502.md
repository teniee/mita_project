# J1 — Railway "Application failed to respond" 502 after idle

## Symptom (first-hand, session-6)
The first request(s) after ~7–8 minutes of no traffic hang ~15 s and return:

```
502 {"status":"error","code":502,"message":"Application failed to respond","request_id":"…"}
```

Observed windows: `/users/me` (×2), `POST /auth/refresh-token`, `POST /onboarding/submit`,
`GET /ai/personalized-feedback`. Recovers within ~15–30 s; subsequent requests are 200.

## Diagnosis — this is Railway EDGE behavior, not the app
Evidence it is generated by Railway's edge (Envoy) proxy and never reaches the app:
- The 502 body is Railway's edge JSON (`"Application failed to respond"` + `request_id`),
  not the app's `standardized_error_middleware` shape.
- **Absent from the app logs** — no matching `Request started/completed` line in the
  uvicorn/Railway service logs for the 502'd requests.
- **No container restart** — uvicorn workers started once (17:43) and stayed up across the
  502 windows (22:51, 22:59). So it is **not** a cold start / worker-startup delay and
  **not** app readiness.
- `/health` (unauthed) warm-up pings succeeded even right before an authed-endpoint 502.

Ruled out:
- **App cold start / worker startup**: no restart in logs.
- **DB pool staleness**: `create_async_engine` already uses `pool_pre_ping=True` (+
  `pool_recycle=1800`), which revalidates/reconnects stale connections.

Most probable cause: **connection-reuse mismatch between Railway's edge and uvicorn.**
uvicorn's default `--timeout-keep-alive` is **5 s**. Railway's edge holds upstream
keep-alive connections longer; when the edge sends a request on a connection uvicorn has
already half-closed, the edge reports a 502 and then re-establishes — matching the
"first request after idle fails, then recovers" pattern.

## Fix applied (app-actionable, NOT client suppression)
`start.sh` / `scripts/deployment/start.sh` now pass `--timeout-keep-alive 75`
(`UVICORN_KEEPALIVE` overridable). Holding the connection open **longer than the edge's
idle window** makes the edge close first, cleanly, so it never sends on a half-closed
socket. This does not hide the error; it removes the connection-reuse race.

> Note: the mobile interceptor already suppresses the *global red banner* only for
> non-core paths (e.g. `/ai/personalized-feedback`); core paths still surface. That is a
> UX decision, unrelated to this server fix, and is not "hiding" the 502.

## Residual — infrastructure owner action (mikhail@mita.finance)
If 502-after-idle still occurs after this deploy, verify on the Railway dashboard:
1. **App Sleeping / serverless**: Settings → confirm the service is NOT set to sleep on
   idle (that would cold-start the container on the next request → genuine 502). If on,
   disable it or set a minimum instance.
2. **Keep-warm**: add a cron ping to `GET /health` every ~4 min (Railway cron or an
   external monitor) so the edge/app connection and DB pool never go fully idle.
3. **Edge timeout**: if long first-request latency persists, confirm the Railway
   proxy request timeout is ≥ the app's worst-case cold-path latency.

Owner action is a dashboard/settings check + optional keep-warm cron; the app-code part
(keep-alive) is done.
