## КРИТИЧЕСКИЕ НЕСООТВЕТСТВИЯ (исправлять первыми)

### ПРОБЛЕМА 1 — Главный API эндпоинт игнорирует движок
- **Файл:** `app/api/budget/routes.py` строки 386-401
- **Что делает:** `POST /budget/monthly` возвращает захардкоженные 30%/15%/10%/20% без вызова `generate_budget_from_answers` или любого реального движка. `user_answers` читается но результат выбрасывается.
- **Что должно:** Вызывать `generate_budget_from_answers(answers)` → `build_monthly_budget()` → вернуть реальный персонализированный бюджет.

### ПРОБЛЕМА 2 — Real-time rebalancing не реализован
- **Файл:** Отсутствует (функция `redistribute_budget_after_overspend` описана в `BUDGET_LOGIC_DEEP_ANALYSIS.md` но в коде не существует)
- **Что делает:** Ничего. При перерасходе оставшиеся дни месяца не пересчитываются.
- **Что должно:** При каждой транзакции которая приводит к перерасходу категории → пересчитывать `planned_amount` для оставшихся дней текущего месяца пропорционально. Это главная заявленная фича продукта.

### ПРОБЛЕМА 3 — Cron перераспределяет прошлый месяц
- **Файл:** `app/services/core/engine/cron_task_budget_redistribution.py` строка 16
- **Что делает:** `month = now.month - 1` — перераспределяет месяц который уже закончился
- **Что должно:** Работать с текущим месяцем в реальном времени, а не постфактум

### ПРОБЛЕМА 4 — Savings не защищены от перераспределения
- **Файл:** `app/services/budget_redistributor.py` строки 41-45
- **Что делает:** При нехватке бюджета берёт из savings так же как из entertainment. Нет иерархии категорий.
- **Что должно:** Никогда не брать из: `savings_goal`, `savings_emergency`, `rent`, `mortgage`, `debt_repayment`, `utilities`, `insurance`. Брать первыми из: `dining_out`, `entertainment`, `hobbies`, `gaming`, `media_streaming`.

### ПРОБЛЕМА 5 — Behavioral allocator сломан (возвращает False True)
- **Файл:** `app/services/core/behavior/behavioral_budget_allocator.py` строки 103-108
- **Что делает:** `UserContext(...)` создаётся но не присваивается переменной. Применяются жёсткие константы food=30%, savings=20% для всех пользователей. API возвращает `"user_context_applied": True` — ложь.
- **Что должно:** Использовать созданный UserContext для персонализации весов через `dynamic_threshold_service`.

### ПРОБЛЕМА 6 — Порог статуса дня одинаков для всех доходов
- **Файл:** `app/services/core/engine/calendar_updater.py` строка 26
- **Что делает:** `delta <= $10.00` → "yellow" для всех пользователей. Для LOW-тира ($50/день) это 20% отклонение. Для HIGH-тира ($500/день) это шум.
- **Что должно:** Использовать `dynamic_threshold_service` — порог пропорционален дневному бюджету пользователя (например: 5% от daily_budget).

### ПРОБЛЕМА 7 — CLUSTERED распределение случайное (без seed)
- **Файл:** `app/services/core/engine/calendar_engine.py` строки 127-133
- **Что делает:** `random.sample()` без seed. При каждой регенерации dining out и entertainment попадают на разные дни. Пользователь не может планировать.
- **Что должно:** Детерминированное распределение. Использовать `user_id + month` как seed.

### ПРОБЛЕМА 8 — Два механизма перераспределения несовместимы
- **Файл A:** `app/engine/budget_redistributor.py` — дневное перераспределение, не сохраняет в БД
- **Файл B:** `app/services/budget_redistributor.py` — категорийное, сохраняет в БД
- **Проблема:** Алгоритм A результат не записывает в DailyPlan. Нет кода связывающего их. Используются в разных контекстах с разными форматами данных.
- **Что должно:** Один единый алгоритм с категорийной иерархией, работающий с DailyPlan в реальном времени.

### ПРОБЛЕМА 9 — Цели изолированы от бюджета
- **Файл:** `app/services/goal_budget_integration.py` — существует но не вызывается из redistribution и alerts
- **Что делает:** Цели живут отдельно. При перерасходе категории нет сообщения "твоя цель [Отпуск] сдвинулась на N дней"
- **Что должно:** `goal_budget_integration` вызывается при каждом перерасходе, каждом перераспределении, каждом alert.

### ПРОБЛЕМА 10 — Уведомления не знают о целях пользователя
- **Файл:** `app/services/budget_alert_service.py`, `app/services/spending_prevention_service.py`
- **Что делает:** Сообщения безличные: "Consider reducing this category". Не знают о целях. Тон нейтральный без мотивации.
- **Что должно:** "Марина, ты потратила на $250 больше на одежду. Твоя цель [Отпуск в Барселоне] сдвинулась на 5 дней. Хочешь скорректировать бюджет?"

### ПРОБЛЕМА 11 — Нет напоминаний через N дней
- **Что делает:** Единственный cron — раз в месяц для перераспределения прошлого месяца. Нет уведомлений-напоминаний.
- **Что должно:** Через 3 дня после того как пользователь проигнорировал предупреждение → напомнить о цели и последствиях перерасхода.

### ПРОБЛЕМА 12 — Несоответствие порогов между сервисами
- **Файл A:** `app/services/budget_alert_service.py:24-25` — WARNING=80%, DANGER=100%
- **Файл B:** `app/services/spending_prevention_service.py:24-27` — CAUTION=70%, DANGER=90%
- **Что должно:** Единые пороги из `dynamic_threshold_service`, единые для всего приложения.

### ПРОБЛЕМА 13 — История перераспределений — эвристика, не лог
- **Файл:** `app/api/budget/routes.py` строки 261-280
- **Что делает:** Сравнивает соседние DailyPlan записи, угадывает причину изменения по размеру diff.
- **Что должно:** Реальный audit log событий перераспределения с причиной, суммой, откуда и куда.

### ПРОБЛЕМА 14 — Перенос излишков savings между месяцами
- **Что делает:** Если в savings накопился surplus за месяц — в следующем месяце он теряется (бюджет строится заново с нуля).
- **Что должно:** Автоматически добавлять surplus savings к цели следующего месяца.

### ПРОБЛЕМА 15 — Нет положительных уведомлений (wins)
- **Что делает:** Система только предупреждает о проблемах. 7 дней подряд в green — ничего.
- **Что должно:** "Ты 7 дней укладываешься в бюджет. Ты на $X ближе к цели [Отпуск]!" — мотивация по науке (Thaler: celebrate small wins).

## ОТСУТСТВУЮЩИЕ КОМПОНЕНТЫ (строить с нуля)

### ОТСУТСТВУЕТ A — Real-time rebalancing engine
При транзакции: если `spent > planned` для категории →
1. Определить overspend = spent - planned
2. Найти оставшиеся дни месяца
3. Найти доноров по иерархии (Level 3 → Level 2, никогда Level 0)
4. Пересчитать `planned_amount` для оставшихся дней
5. Сохранить в DailyPlan
6. Вернуть пользователю уведомление с контекстом цели

### ОТСУТСТВУЕТ B — Goal-aware notifications
Каждый alert должен включать:
- Имя пользователя
- Название категории с перерасходом
- Сумма перерасхода
- Влияние на цель (в днях/рублях/процентах)
- Конкретное предложение (откуда взять деньги)
- Кнопки действий: [Принять] [Настроить] [Игнорировать]

## АРХИТЕКТУРНЫЕ РЕШЕНИЯ

- Savings СВЯЩЕННЫ — это смысл приложения. Никогда не доноры.
- MITA — советник, не судья. Конечный выбор всегда за пользователем.
- Каждое уведомление = имя + категория + цель + предложение + действие.
- Напоминание если проигнорировал = через 3 дня.
- Пороги всегда из `dynamic_threshold_service` (пропорционально доходу).
